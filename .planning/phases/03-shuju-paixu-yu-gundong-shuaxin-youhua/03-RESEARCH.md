## 2. 需求分析

### 2.1 SORT-01: 卡片排序问题

**问题**: 最新数据应前置显示

**现状**:
- 后端已按 `time` 降序排列 ✅
- 需验证前端是否正确渲染排序结果

**需要确认**: 
- 数据中 `time` 字段的来源和格式（Unix时间戳？字符串？）
- 是否存在 time 为空或 0 的数据导致排序不稳定

### 2.2 SORT-02: 滚动游标刷新无反应

**问题**: 用户滚动时无法加载更多数据

**现状**:
- `setupLazyLoading()` 只处理图片懒加载（第248-265行）
- 没有处理列表无限滚动

**需要实现**:
- 无限滚动触发器（距离底部 100px）
- 分页加载逻辑（使用 offset/limit）
- 加载状态指示器

### 2.3 SORT-03: 排序方式切换 UI

**问题**: 添加按时间/按热度排序选项

**现状**:
- 无排序选择器

**需要实现**:
- 排序下拉菜单 UI
- 时间排序（已实现）
- 热度排序（按 liked_count 排序）
- localStorage 持久化

## 3. 技术实现建议

### 3.1 无限滚动实现

```javascript
// 方案：使用 IntersectionObserver 监听底部 sentinel 元素
const sentinel = document.createElement('div');
notesGrid.appendChild(sentinel);

const observer = new IntersectionObserver((entries) => {
    if (entries[0].isIntersecting && !loading && hasMore) {
        loadMoreNotes();
    }
}, { rootMargin: '100px' });
```

### 3.2 排序功能实现

```javascript
// 方案：后端增加排序参数
// GET /notes?sort=time&order=desc

// 或前端排序（仅适用于小数据量）
notes.sort((a, b) => b.time - a.time);
```

### 3.3 API 扩展

现有 API 已支持分页：
- `offset`: 起始位置
- `limit`: 返回数量

**可选扩展**:
- 添加 `sort` 参数支持后端排序（热度排序需要）

## 4. 风险与依赖

### 4.1 潜在风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 数据量大时前端排序慢 | 用户体验 | 改用后端排序 |
| time 字段为空/0 | 排序不稳定 | 使用 stable sort 或secondary key |
| 无限滚动触发频繁 | API 压力 | 添加 debounce 和 loading 状态 |

### 4.2 技术依赖

- **后端**: 无需大改，已有分页支持
- **前端**: 需要新增无限滚动和排序 UI
- **样式**: 需要加载更多指示器样式

## 5. 决策要点

在规划实现方案前，需要明确：

1. **排序策略**: 使用前端排序还是后端排序？
   - 数据量 < 1000：前端排序即可
   - 数据量 > 1000：需要后端支持

2. **热度排序需求**: 是否必须在 Phase 3 实现？
   - 03-CONTEXT.md 中标记"暂不实现"
   - 若需要，API 需扩展排序参数

3. **无限滚动范围**: 是否应用于所有平台？
   - 当前 app.js 是小红书专用
   - 抖音/B站/知乎需分别实现或抽离公共逻辑
