# 小红书内容爬取系统

## Problem Frame

用户需要从小红书平台检索特定主题的笔记，爬取文案和图片内容，以便搬运到其他平台。传统爬虫方案面临以下挑战：

- 小红书有严格的反爬机制（签名算法、登录态检测、浏览器指纹识别）
- 普通请求方式极易被封禁
- 逆向签名算法成本高，平台更新后需重新逆向
- 需要稳定、安全的方案，而非追求高效率

用户的核心诉求是**稳定可靠**地获取内容，而非高频批量爬取。

## Requirements

**登录与认证**
- R1. 支持二维码扫码登录小红书
- R2. 支持登录态缓存，避免重复扫码

**内容爬取**
- R3. 支持关键词搜索小红书笔记
- R4. 爬取笔记的文案内容（标题、正文、标签）
- R5. 下载笔记中的图片（原图或高清图）
- R6. 支持爬取笔记的基本信息（点赞数、收藏数、评论数等）

**数据存储**
- R7. 将爬取的内容保存到本地
- R8. 支持结构化存储（JSON格式）
- R9. 图片按笔记分类存储，便于后续处理

**稳定性与安全**
- R10. 采用浏览器自动化方案，无需逆向签名算法
- R11. 模拟真实用户行为，避免触发反爬机制
- R12. 支持请求频率控制，降低封禁风险

## Success Criteria

- 能够成功登录小红书并保持登录态
- 能够通过关键词搜索并获取笔记列表
- 能够完整爬取笔记文案和图片
- 连续运行 72 小时以上不被封禁
- 数据结构清晰，便于后续处理

## Scope Boundaries

- 本系统仅爬取公开数据，不涉及私密内容
- 不支持自动化搬运到其他平台（仅本地存储）
- 不支持视频内容的下载（仅图片和文案）
- 不支持评论区的深度爬取（可后续扩展）
- 不支持多账号管理（可后续扩展）
- 不追求高频爬取，稳定性优先

## Key Decisions

### 技术选型：MediaCrawler

**决策**: 采用 MediaCrawler 开源项目作为核心框架

**理由**:
1. 成熟稳定：46K+ GitHub Stars，社区活跃
2. 无需逆向：基于 Playwright 浏览器自动化，不依赖签名算法逆向
3. 功能全面：已支持小红书关键词搜索、登录态缓存、图片下载等核心功能
4. 易于使用：提供 WebUI 界面，降低使用门槛
5. 持续更新：作者活跃，平台反爬更新后快速适配

**技术原理**:
```
用户 → Playwright 浏览器 → 模拟扫码登录 → 获取登录态
                                    ↓
                          执行 JS 获取签名参数
                                    ↓
                          发起 API 请求获取数据
                                    ↓
                          解析并保存到本地
```

### 登录方式：二维码扫码

**决策**: 采用二维码扫码登录

**理由**:
- 用户体验好：扫码即可，无需手动输入
- 安全性高：不存储账号密码
- 实现简单：MediaCrawler 原生支持
- 登录态缓存：一次扫码长期有效

### 爬取方式：关键词搜索

**决策**: 采用关键词搜索模式

**理由**:
- 符合用户需求：检索特定主题的笔记
- 灵活性高：可随时更换搜索关键词
- 覆盖面广：可发现新内容

### 存储方式：本地文件存储

**决策**: 采用本地文件存储（JSON + 图片文件夹）

**理由**:
- 简单直接：无需额外配置数据库
- 便于处理：JSON 格式易于解析
- 可扩展：后续可对接其他系统
- MediaCrawler 原生支持

## Dependencies / Assumptions

### 依赖
- Python 3.8+
- Node.js 16+（部分平台需要）
- Playwright 浏览器驱动
- MediaCrawler 项目代码

### 假设
- 用户有稳定的小红书账号
- 网络可正常访问小红书
- 爬取频率较低（不需要代理池）
